1. PROJECT PURPOSE
──────────────────
ERP Core is a centralized Educational Resource Planning (ERP) platform for academic
institutions. It provides four distinct, fully authenticated portal experiences:

  • Students    — Academic health, attendance, schedule, assignments, GPA trends
  • Staff       — Course management, leave approvals, mentee tracking, daily schedule
  • Admins      — System-wide oversight, user directory, audit logs, MFA status
  • Management  — Executive KPIs, financial health, department analytics, admissions

All portals share a unified authentication layer with role-based redirection, MFA
verification, and a persistent theme system (light/dark).

4. AUTHENTICATION FLOW
────────────────────────
Route: /   → RolePortal → SignInCard

  Step 1 — Role Selection
    User picks a role via the four-tab switcher: Student | Staff | Management | Admin.
    Each role has its own gradient color and icon on the card.

  Step 2 — Credentials
    Email + Password submitted to POST /login (or POST /signup for new accounts).
    Backend validates and returns a userId + mfaCode (displayed via toast notification).

  Step 3 — MFA Verification
    User enters the 6-digit OTP in InputOTP. Submitted to POST /verify-mfa.
    On success: JWT-style session stored in localStorage (userRole, userId).
    Redirect to /dashboard.

  Step 4 — Role Dispatch (DashboardRouter in App.tsx)
    Reads localStorage userRole and renders the matching portal:
      student    → <Dashboard />
      staff      → <StaffDashboard />
      admin      → <AdminDashboard />
      management → <ManagementDashboard />

  Forgot Password: Route /forgot-password → <ForgotPasswordCard />

5. PORTAL PAGES & SECTIONS

All portals share the same structural pattern:
  • Collapsible sidebar (desktop, 240px expanded / 72px icon-only)

  • Sections without a dedicated render branch show a placeholder card with
    the section icon, "coming soon" message, and "Back to Dashboard" button

6. SHARED UI/UX PATTERNS
──────────────────────────
Three-State Law: All data components handle Loading (Skeleton) → Success (data)
                 → Empty/Error states. No blank rendering.

Optimistic UI:   Staff leave approvals update immediately in UI, then syncs to
                 backend. Failure would roll back (toast notification shown).

Theme system:    ThemeProvider wraps all dashboard routes. Login page uses a fixed
                 hardcoded dark theme (deep blue gradient). No system-level theme
                 interference on the auth pages.

Notifications:   Sonner toast system used throughout. Variants: success (green),
                 error (red), info (blue), warning (amber). MFA codes delivered
                 via toast + optional browser push notification.

Navigation:      Mobile-first. Sidebar hidden on mobile with hamburger. Clicking
                 any nav item updates activeSection and closes the drawer.
